## Summary

Phase 05.1 should integrate providers behind a single server-side adapter, not directly inside route handlers or UI. The existing service/orchestrator split is already the right insertion point: replace `buildDefaultProposal` behavior in `createAdaptiveCoachingService` with a provider-backed `proposeRecommendation` implementation, while leaving the fixed SAFE pipeline untouched.

The strict output strategy should be provider-agnostic at the domain boundary: define one canonical JSON Schema for `AdaptiveRecommendationProposal` and always validate with existing Zod parse helpers (`parseAdaptiveRecommendationProposal`). OpenAI should use strict JSON schema response formatting. Anthropic fallback should use strict tool-schema output (forced tool call) to guarantee schema-valid JSON input payloads.

Runtime reliability should be explicit and short-lived: one bounded attempt on OpenAI, one bounded fallback attempt on Anthropic, then deterministic SAFE-03 behavior. Do not rely on provider SDK default retry behavior alone; it can be too permissive for this phase policy.

**Primary recommendation:** Implement `LlmProposalClient` abstraction with provider adapters (OpenAI primary, Anthropic fallback), canonical strict schema at the boundary, and deterministic timeout/retry/fallback orchestration before current SAFE pipeline.

### Pattern 1: Provider Abstraction With Deterministic Fallback
**What:** One interface returns normalized raw proposal payload + metadata regardless of provider.
**When to use:** Every generation request when `LLM_REAL_PROVIDER_ENABLED=true`.
**Example:**
```typescript
export type LlmProposalClient = {
  generate(input: ProposalPromptInput): Promise<{
    proposal: unknown;
    meta: { provider: 'openai' | 'anthropic'; model: string; latencyMs: number; requestId?: string };
  }>;
};
```

### Pattern 2: Canonical Schema Once, Provider Mapping Per Adapter
**What:** Keep one domain schema (`AdaptiveRecommendationProposal`) and map to provider-specific strict schema formats.
**When to use:** Before each provider call and after each provider response.
**Example:**
```typescript
// 1) Send provider strict schema
// 2) Receive JSON payload
// 3) Parse with parseAdaptiveRecommendationProposal(payload)
```

### Pattern 3: Timeout/Retry/Fallback Ordering
**What:** Enforce runtime policy outside provider defaults.
**When to use:** All provider invocations in phase 05.1.
**Recommended order:**
1. OpenAI attempt with strict timeout.
2. If timeout/network/5xx/429/schema-invalid/refusal: one immediate bounded retry on OpenAI.
3. If still failed: Anthropic fallback attempt.
4. If Anthropic fails or schema invalid: return `null` proposal and let existing SAFE-03 path resolve conservative fallback.

### Pattern 4: Non-PII Observability Envelope
**What:** Log only operational metadata, never user free text or profile payloads.
**When to use:** One audit log per attempt and one rollup log per request.
**Fields:** `provider`, `model`, `attempt`, `timeoutMs`, `latencyMs`, `outcome`, `parseStatus`, `fallbackReasonCode`, `requestId`, `traceSteps`.

### Anti-Patterns to Avoid
- Embedding provider SDK calls directly in route handlers.
- Trusting model `status`; status must remain server-assigned.
- Parsing free-text responses with regex.
- Logging prompt body, profile details, limitations text, or recommendation reasons verbatim (PII/sensitive drift risk).

## Strict Structured Output Strategy

### OpenAI (Primary)
- Use Structured Outputs with strict JSON schema (`response_format`/`text.format` JSON schema + `strict: true`).
- Build schema to satisfy strict constraints (`additionalProperties: false`, all fields required; optionality via nullable union).
- Handle explicit refusal/incomplete output as invalid proposal for retry/fallback flow.

### Anthropic (Fallback)
- Use Messages API tool schema (`input_schema`) for output shaping.
- Force tool invocation with `tool_choice: {"type":"any"}` and set `strict: true` on tool definition for schema conformance.
- Use one synthetic "emit_adaptive_recommendation" tool so returned tool input is the proposal JSON.

### Domain Enforcement (Provider-Agnostic)
- Always run `parseAdaptiveRecommendationProposal` after provider response.
- Additional phase rule: reject proposal if `evidenceTags` missing/empty.
- Ignore model-provided `status` on persistence path; recompute in orchestrator.

## Timeout, Retry, and Fallback Policy

Policy for 05.1 (prescriptive):
- App-level retries: `1` additional attempt max on primary provider only.
- Fallback provider: `1` attempt max.
- Per-attempt timeout: `4.5s` to `5s` (strict short budget).
- Overall budget: hard cap `10s` to avoid blocking adaptation endpoint.

Failure types that trigger fallback chain:
- Timeout/network transport errors.
- HTTP 408/409/429/5xx (or provider overloaded equivalents).
- Structured output parse failure (including missing `evidenceTags`).
- Provider refusal or incomplete response.

SAFE-03 handoff rule:
- If no valid proposal after chain, pass `null` candidate downstream so existing `safe_03` logic chooses `reuse_last_conservative` or `conservative_hold`.

## Don’t Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Provider retry/backoff machinery | Custom retry framework | SDK retry controls + explicit app-level orchestration | Faster, less bug surface, clearer policy caps. |
| Provider-native domain schema | Separate schema per provider | One canonical domain schema + provider adapters | Prevents drift and inconsistent behavior. |
| Ad-hoc log masking | Manual string redaction everywhere | Structured allowlist audit fields | Easier to prove non-PII compliance. |

**Key insight:** Reliability here is mostly orchestration discipline and schema discipline, not model prompting complexity.

## Common Pitfalls

### Pitfall 1: Double-Retry Explosion
**What goes wrong:** SDK default retries plus app retries create excessive latency.
**How to avoid:** Set explicit per-provider retry config and compute total attempt ceiling.

### Pitfall 2: "Valid JSON" but Invalid Contract
**What goes wrong:** JSON parses but violates domain requirements (`evidenceTags` missing, wrong enums).
**How to avoid:** Always parse through domain Zod contract, not `JSON.parse` success alone.

### Pitfall 3: Fallback Drift From SAFE-03
**What goes wrong:** Provider fallback produces behavior that bypasses existing SAFE policy.
**How to avoid:** Provider layer only supplies proposal candidate; SAFE policy remains sole authority for final recommendation.

### Pitfall 4: Observability Leaks
**What goes wrong:** Prompt/user payload logged during incident debugging.
**How to avoid:** Log only structured non-PII metadata + request IDs.

## Code Examples

### Provider orchestrator skeleton
```typescript
export async function proposeWithFallback(input: ProposalPromptInput): Promise<unknown | null> {
  const primary = await tryProvider('openai', input, { retries: 1, timeoutMs: 4500 });
  if (primary.ok) return primary.proposal;

  const secondary = await tryProvider('anthropic', input, { retries: 0, timeoutMs: 4500 });
  if (secondary.ok) return secondary.proposal;

  return null; // existing SAFE-03 path handles conservative fallback
}
```
