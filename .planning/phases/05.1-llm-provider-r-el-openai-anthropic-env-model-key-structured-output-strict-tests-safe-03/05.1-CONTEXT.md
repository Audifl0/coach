<decisions>
## Implementation Decisions

### Provider et mode de decision
- Provider principal en 05.1: **OpenAI**.
- Provider secondaire: **oui**, fallback provider actif des la 05.1 (Anthropic en secours).
- Selection modele/provider via variables d'environnement explicites (pas hardcode runtime).
- Le LLM propose une **recommandation complete** (action + exos/reps/charge) puis le moteur SAFE serveur conserve l'autorite finale.

### Sortie structuree et robustesse
- Sortie LLM: **JSON strict obligatoire** (pas de parsing permissif texte libre).
- En cas de sortie invalide: **1 retry cadre**, puis fallback **SAFE-03**.
- `evidenceTags` requis dans la proposition; absence de tags => sortie invalide.
- Le champ `status` n'est pas accepte tel quel depuis le LLM: **status calcule cote serveur** apres guardrails/confirmation.

### Configuration et exploitation
- Set d'env minimum: **complet provider-ready** (provider, modele, cles, timeout, retry, feature flag).
- Politique runtime: **timeout strict + retry court** puis SAFE-03.
- Observabilite: **audit metier sans PII** (provider, modele, latence, parse status, fallback reason).
- Activation provider reel via feature flag explicite: `LLM_REAL_PROVIDER_ENABLED=true`.

### Compatibilite decisions Phase 05
- Conserve les decisions de 05: LLM-first, guardrails SAFE-01/02/03 obligatoires, confirmations ADAP-03, explications utilisateur traçables.
- Le fallback SAFE-03 reste la voie de securite ultime en cas d'erreur provider, timeout, ou sortie hors schema.

### Claude's Discretion
- Nommage exact des variables d'environnement (tout en restant explicites et documentees).
- Parametrage fin timeout/retry par provider tant que la politique reste stricte.
- Format exact des logs d'audit non-PII.

</decisions>
