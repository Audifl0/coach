## Summary

Phase 05.2 should industrialize the existing `refresh-corpus.ts` pattern into a deterministic weekly pipeline with explicit stage boundaries: source discovery, ingestion, synthesis, quality validation, and atomic publish/rollback. The implementation should remain file-versioned in `.planning/knowledge/adaptive-coaching/` and keep the adaptive runtime as final decision authority (already established in phase 05/05.1).

Given the locked source perimeter, the standard stack should prioritize official metadata APIs (PubMed/NCBI E-utilities, Crossref REST, OpenAlex REST) and avoid ad-hoc scraping. Use source-aware rate/concurrency policies, strict whitelist controls, and robots compliance for any non-API fetch path. Synthesis can reuse the existing strict structured-output discipline introduced in 05.1 to produce principle-level FR summaries with provenance, while keeping EN ingestion inputs.

Publish only from a fully validated candidate snapshot. Never mutate active corpus files in-place. Generate candidate artifacts in a temporary run directory, validate quality/coherence, then atomically promote via `rename`/pointer swap and retain previous snapshots for rollback.

**Primary recommendation:** Build a stage-based pipeline around `scripts/adaptive-knowledge/refresh-corpus.ts` with official API connectors, strict schema+quality gates, and atomic snapshot promotion (`candidate -> validated -> active`) plus one-command rollback.

## Architecture Patterns

### Recommended Project Structure
```text

.planning/knowledge/adaptive-coaching/
├── snapshots/
│   └── 2026-03-05T12-00-00Z/
│       ├── index.json
│       ├── principles.json
│       ├── sources.json
│       └── run-report.json
├── active.json                   # pointer to active snapshot metadata
└── rollback.json                 # pointer to previous snapshot metadata
```

### Pattern 1: Stage-Based Pipeline with Idempotent Checkpoints
**What:** Split run into deterministic stages (`discover -> fetch -> synthesize -> validate -> publish`). Persist stage outputs in run directory.
**When to use:** Every weekly cron execution.
**Why:** Supports bounded retries and per-source skip without corrupting active corpus.

### Pattern 2: Source Adapter Contract
**What:** Normalize every connector output into one internal `NormalizedEvidenceRecord` schema before synthesis/quality.
**When to use:** Immediately after each API response.
**Contract fields:** `source_id`, `source_type`, `url`, `title`, `published_at`, `abstract_or_summary`, `evidence_level_hint`, `tags`, `language`.

### Pattern 3: Composite Quality Gate Before Publish
**What:** Compute quality score from source class weight + recency + contradiction checks + required-field completeness.
**When to use:** After synthesis, before artifact promotion.
**Publish rule:** any critical contradiction or score below threshold => no publish; keep previous snapshot active.

### Pattern 4: Atomic Snapshot Promotion
**What:** Write candidate snapshot under temp directory, validate checksums and schema, then atomically switch active pointer.
**When to use:** Only after full validation pass.
**Implementation note:** Use same-filesystem `rename` semantics and never partial overwrite of active files.

### Pattern 5: Runtime Decoupling
**What:** Adaptive runtime reads `active` snapshot only; pipeline writes to snapshot workspace only.
**When to use:** Always.
**Why:** Prevents runtime from seeing transient/partial writes.

### Anti-Patterns to Avoid
- Directly editing `.planning/knowledge/adaptive-coaching/index.json` in place during fetch.
- Mixing source retrieval, synthesis, and publish logic in one long function.
- Trusting LLM synthesis output without schema + contradiction validation.
- Blocking entire run because one source endpoint fails.

| Problem | Don’t Build | Use Instead | Why |
|---|---|---|---|
| Scholarly metadata discovery | Custom site scraping of journals | Official APIs (NCBI, Crossref, OpenAlex) | Better stability, explicit quotas, legal clarity. |
| Structured output contract | Regex/heuristic parsing of model text | Existing strict JSON schema + Zod parse discipline | Prevents silent schema drift. |
| Atomic publish behavior | Multi-file best-effort copy | `rename`-based snapshot promotion and pointer swap | Avoids partial state exposure. |
| robots policy parsing | Ad-hoc robots rules | RFC 9309-compliant behavior (or strict API-only mode) | Prevents accidental non-compliance. |

**Key insight:** Most phase risk is operational correctness (rate limits, partial failures, atomic publication), not model prompt quality.

## Common Pitfalls

### Pitfall 1: Rate-limit collapse under weekly cron
**What goes wrong:** Burst calls trigger 429/403 and degrade corpus freshness.
**Why it happens:** Missing per-source throttling/concurrency budgets.
**How to avoid:** Source-specific client policies (Crossref polite mode + concurrency cap, OpenAlex budget awareness, NCBI request pacing).
**Warning signs:** Frequent 429, rapidly exhausted OpenAlex daily budget.

### Pitfall 2: Publishing low-confidence or contradictory principles
**What goes wrong:** Runtime receives unstable guidance; safety rationale drifts.
**Why it happens:** Weak contradiction detection or no hard publish gate.
**How to avoid:** Hard failure gate for critical contradiction and score threshold.
**Warning signs:** Frequent conflict flags in run report; sudden principle churn week-to-week.

### Pitfall 3: Partial snapshot activation
**What goes wrong:** Runtime reads mixed old/new artifacts.
**Why it happens:** In-place writes to active files.
**How to avoid:** Candidate directory + atomic pointer swap only after complete validation.
**Warning signs:** `generated_at` mismatches between `index.json` and `principles.json`.

### Pitfall 4: Source perimeter drift
**What goes wrong:** Out-of-whitelist domains enter corpus.
**Why it happens:** Connector fallback paths or unvalidated redirects.
**How to avoid:** Strict domain allowlist check at normalized-record stage.
**Warning signs:** run-report includes unknown domain, missing source classification.

### Pitfall 5: LLM synthesis overreach
**What goes wrong:** Synthesized principles claim unsupported guidance.
**Why it happens:** Insufficient provenance enforcement.
**How to avoid:** Require explicit source links/IDs per principle and min-source-count per principle class.
**Warning signs:** principle with empty provenance or single low-quality source.

## Code Examples

### Stage runner skeleton
```typescript
export async function runCorpusPipeline(now: Date) {
  const runId = now.toISOString().replace(/[:.]/g, '-');
  const runDir = path.join('.planning/knowledge/adaptive-coaching/snapshots', runId);

  const discovered = await discoverSources();
  const fetched = await fetchAllWithBudgets(discovered);
  const synthesized = await synthesizePrinciplesStrict(fetched);
  const validation = validateCandidate({ fetched, synthesized });

  if (!validation.publishable) {
    await writeRunReport(runDir, { status: 'blocked', validation });
    return { published: false, reason: validation.reason };
  }

  await writeSnapshot(runDir, synthesized);
  await promoteSnapshotAtomically(runDir);
  return { published: true, snapshot: runId };
}
```

### Source-aware retry/skip
```typescript
async function fetchWithBoundedRetry<T>(label: string, fn: () => Promise<T>, maxRetries = 2): Promise<T | null> {
  for (let i = 0; i <= maxRetries; i += 1) {
    try {
      return await fn();
    } catch (error) {
      if (i === maxRetries) {
        logger.warn({ stage: 'fetch', source: label, error: String(error) }, 'source skipped after retries');
        return null;
      }
      await wait(400 * (i + 1));
    }
  }
  return null;
}
```

### Atomic pointer swap
```typescript
async function promoteSnapshotAtomically(snapshotDir: string) {
  const activeTmp = '.planning/knowledge/adaptive-coaching/active.tmp.json';
  const active = '.planning/knowledge/adaptive-coaching/active.json';

  await writeFile(activeTmp, JSON.stringify({ snapshotDir, promoted_at: new Date().toISOString() }) + '\n', 'utf8');
  await rename(activeTmp, active); // same filesystem atomic replace
}
```

## Open Questions

1. **Exact quality threshold values**
- What we know: thresholding is explicitly discretionary.
- What’s unclear: numeric cutoff per source class and contradiction severity.
- Recommendation: set initial conservative defaults in planning (`publish_threshold`, `critical_contradiction_block=true`) and tune after 2-3 weekly reports.

2. **Synthesis language pipeline**
- What we know: EN inputs, FR output for coach.
- What’s unclear: whether FR synthesis is generated directly or translated post-synthesis.
- Recommendation: keep normalized evidence in EN, synthesize principles in FR with mandatory EN provenance fields.

3. **Active corpus loading strategy in runtime**
- What we know: runtime currently uses in-code default corpus fallback patterns.
- What’s unclear: whether loading from snapshot should happen at boot or per-request cached read.
- Recommendation: boot-time load with periodic refresh cache; fail closed to last known valid snapshot.

4. **Cron execution location**
- What we know: trigger must be cron-only in nominal mode.
- What’s unclear: host cron vs container cron vs external scheduler.
- Recommendation: host-level cron invoking project script for simplest VPS ops and clear ownership.
